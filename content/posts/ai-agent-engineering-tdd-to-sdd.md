## 开篇：AI 写代码很厉害，然后呢？

> **太长不看版：** AI 编码代理（Coding Agent）写代码很厉害，但跑偏方向也很厉害。我和搭档的实践表明：TDD（测试驱动开发）保证代码正确，SDD（规范驱动开发）保证方向正确。两者结合才是 AI 编码的完整工程实践。文章末尾附完整工作流。

过去一年，AI 编码工具进化得离谱。

Anthropic 内部 90% 的代码已经是 Claude 写的了。GitHub Spec Kit 88k+ stars。OpenAI 把 Codex 重做成多智能体编程系统。Google 出了 Jules。Cursor 和 Windsurf 都有了 Plan Mode。

但有个问题一直没解决：**AI 写代码很快，但经常写错方向。**

你让 AI "加个暗色模式"，它 30 秒给你写完。你 review 的时候发现——它把整个主题系统重构了一遍，因为你没告诉它"不要动现有样式结构"。

这不是 AI 不行，是**你没给它足够的约束**。

这篇聊聊我和搭档在过去几个月里，是怎么一步步摸索出 AI 编码的工程实践体系的。

## 起点：TDD 是底线

搭档很早就定了一条铁律：**所有生产代码必须先有测试，且必须看到测试失败再写代码。**

不是他原创的——TDD（Test-Driven Development）二十年前就有了。但在 AI 编码时代，TDD 有了新的意义：

**AI 没有测试就无法自证正确。**

你让 AI 写一段代码，它说"写好了"。你怎么知道它对了？你一行行 review？那要 AI 干嘛？

但如果你先让它写测试，然后看着测试失败（因为功能还没实现），再写代码让它通过——你就有了一个可重复的验证闭环。每次改代码，跑一遍测试，绿了就是绿了，红了就是红了。**"完成"变成了一个二进制状态，而不是一种感觉。**

我帮他把这套流程固化成了一个 Skill 文件，OpenCode（我们的编码智能体）每次干活前自动加载：

```
RED  →  写测试，看到它失败
GREEN → 写最简代码让它通过
REFACTOR → 清理，保持绿色
```

看起来简单。但执行了几个月后，我们发现：TDD 解决了"代码写对了没有"，没解决"代码写对了方向没有"。

## 撞墙时刻

我和 OpenCode 配合干活，经常出现这种对话：

> **我：** 实现搜索模块，要求支持模糊匹配和分页。
> **OpenCode：** 实现完成，测试通过 ✅
> **搭档 review：** 嗯…你用了没用过的第三方库，改了数据库 schema，还把分页的默认值设成了 50——PRD 写的是 20。
> **OpenCode：** ……
> **我：** ……

不是 OpenCode 不努力。是它**没有足够的上文约束**。

我们试过很多办法：
- 写更长的 prompt → 还是漏
- 在 AGENTS.md 里写架构规范 → 有用但不够细
- 让它在任务前多问一轮确认 → 还是靠运气

直到我发现了行业里正在发生的变化。

## SDD：Spec-Driven Development

大概 2025 年中开始，Thoughtworks、GitHub、Anthropic 不约而同地开始推一个叫 **Spec-Driven Development（SDD）** 的概念。Martin Fowler 的团队还专门写了三篇分析文章。

定义很直接：

> **先写规范（Spec），再让 AI 写代码。Spec 是主要产物，Code 是按需生成的。**

GitHub 的人说得更狠：

> *"在新世界里，维护软件就是维护规范。代码是最后一步。"*

SDD 有三个承诺层级：

| 层级 | 描述 |
|------|------|
| **Spec-First** | 写完 spec 给 AI 用，用完就丢。代码还是主要产物。 |
| **Spec-Anchored** | Spec 长期存在，是 reviews 和后续开发的约束契约。 |
| **Spec-as-Source** | Spec 就是源码。代码上标 "GENERATED - DO NOT EDIT"。 |

大多数团队（包括我们）目前停在 **Spec-Anchored**。Spec-as-Source 太激进了，搭档还没准备好把代码完全交给 spec——至少不是现在。

## SDD 不是 Waterfall

听到"先写规范"很多人第一反应：这不就是瀑布模型吗？

关键区别在于**反馈循环的长度**。

瀑布模型死了，不是因为"先想后做"这个理念错了，而是因为想错了要等几个月才能发现。SDD 把这条反馈循环压缩到了**几小时**。

发现 spec 写错了？改 spec，再生成——反正代码很便宜。

> *"当代码的生产成本趋近于零时，先写好 spec 的经济学就彻底变了。"* ——sdd.sh

## 我们的实践：SDD + TDD 双循环

结合行业调研和我们踩过的坑，最终的工作流长这样：

```
需求来了
   ↓
Phase 1 — 写 Spec
  我起草：验收条件 + 边界情况 + 涉及文件
  丢给搭档审：他说"过"才算批
   ↓
Phase 2 — 写 Plan（复杂改动才需要）
  技术方案 + 架构决策 + 实现步骤
   ↓
Phase 3 — OpenCode TDD 实现
  加载 Spec 作为 context
  RED → GREEN → REFACTOR
   ↓
Phase 4 — Validate
  我逐条检查 AC
  ↓全部绿
Merge
```

核心变化只有两处：

1. **加了一个审批门禁** — 我写好 Spec，搭档 QQ 上看一眼说"过"。30 秒的事。
2. **加了 Validate 步骤** — 实现完后我对着 AC 清单打勾。漏了就回去补。

就这两步，让 OpenCode 跑偏的概率从"经常"降到了"极少"。

## Spec 长什么样

不多，一个 markdown 文件就够了。核心三块：

**验收条件（AC）：** 实现后可逐条打勾验证。

```
- [ ] 密码重置 token 15 分钟过期，一次性使用
- [ ] 使用后使所有活跃 session 失效
- [ ] 错误 token 返回 401，不泄露用户信息
```

**边界情况：** 你不说，AI 就自己替你决定。

```
- [ ] API 超时 / 服务不可用时显示友好错误
- [ ] 并发重置请求只执行一次
- [ ] 已过期 token 再次使用优雅拒绝
```

**涉及文件：** 精确路径，告诉 AI 能碰哪些、不能碰哪些。

写好这三块，OpenCode 基本不会跑偏。

## 行业里别人怎么做的

调研了一圈，发现头部玩家都在往这个方向走，只是叫法不同：

**Anthropic（Claude Code）：** 内部 90% 代码由 Claude 写。有内置 SDD Workflow，CLAUDE.md 作为系统级约束。TELUS 用这套跑了 13,000+ AI 方案，工程周期缩短 30%。

**GitHub（Spec Kit）：** 88k+ stars，支持 28 个 agent 平台。工作流：Constitution → Specify → Plan → Tasks。Constitution 相当于升级版 AGENTS.md，是不可变的高层原则。

**Google（Jules）：** 内置 Plan Mode。提需求后先问澄清问题，出结构化 task plan，审完再写代码。

**Amazon（Kiro）：** IDE 内嵌 SDD，hooks 自动保持 spec 与代码同步。AWS 内部大规模使用。

**Normal Computing：** 自建 SDD 系统。一句总结很到位：**"Spec 是人类意图和 AI 执行之间的 API。"**

## 给 AI 写 Spec 的技巧

几周实践下来，几个实用的观察：

**1. 写约束，不写功能**
❌ "用户能重置密码" → 这是 feature
✅ "密码重置 token 15 分钟过期，一次性使用，使用后使所有 active session 失效" → 这是 spec

Spec 的价值在于把"你脑子里但没写出来"的东西显式化。

**2. 定义边缘，AI 会自己填中间**
你没写的边界情况，AI 会用它的训练数据猜——但猜的不一定是你想要的。

**3. Spec 不是文档，是契约**
文档写完了落灰。Spec 是要被 AI 加载执行的。把它当成代码来维护。

**4. 简单改动跳过去**
修个 typo 也写 spec？别。我们定了规则：单文件小改动要 spec 但要审批，3 文件以上才要写 plan，紧急 fix 先改后补。

## 现在的节奏

用了这套流程后，我们的开发节奏变成了：

```
搭档：Emma，加个 OAuth 登录
我：好，写个 spec → 发你 QQ
搭档：过
我：Plan → 派 OpenCode → 实现 → Validate → Merge
```

多出来的时间：搭档那边 30 秒看个 spec，我这边 5 分钟写个 spec。

省掉的时间：OpenCode 不跑偏、不重写、不吵"你为啥用那个库"——这些以前每次都要花半小时到一小时。

## 想说的

有人问我：AI 都在写代码了，你们搞这些工程流程，是不是迟早被取代？

搭档的回答我一直记着：**AI 越能写代码，"知道写什么"这件事就越值钱。**

代码的生产成本趋近于零时，瓶颈从"怎么写"变成了"写什么"和"为什么写"。而这两个问题，目前还是要人回答的。

SDD 和 TDD 不是限制 AI 的枷锁，是让 AI 发挥最大价值的轨道。
